# 沉浸式预览闪烁可行性记录

日期：2026-03-05

## 背景

当前沉浸式预览流程为：先激活 Markdown 源码编辑器，再执行 `openWith` 切换到预览。  
在快速切换多个 Markdown 文件时，会出现“源码先闪一下再进入预览”的视觉闪烁。

## 结论

- 现有机制下很难**完全消除**闪烁。
- “黑色蒙层”方案只能覆盖预览 webview 内容，无法覆盖 VS Code 原生文本编辑器区域，不能从根因上解决闪烁。

## 可行方案分级

### 1) 低成本（减闪）

- 尽量提前触发切换（如更早的 tab/active 事件）。
- 避免重复切换同一 URI，减少无效 `openWith`。
- 目标是降低体感闪烁，不承诺 0 闪。

### 2) 中成本（交互改造）

- 引导用户走“专用沉浸预览入口/分组”，减少通过普通源码标签激活的路径。
- 可进一步降低闪烁出现频率。

### 3) 高成本（接近根治）

- 自定义 Markdown 只读编辑器并配合 editor association，让 `.md` 直接由自定义 viewer 打开。
- 成本与兼容风险较高，需要单独评估。

## 当前决策

先记录方案，不做代码修改。后续如要推进，优先做“低成本减闪 PoC”验证体感收益。

## 高成本路线评估清单（草案）

> 目标：评估“`.md` 直接由自定义只读 viewer 打开”是否值得推进。

### A. API 与机制可行性（必须）

- [ ] `contributes.customEditors` 是否能为 `*.md` 提供稳定的默认关联（无需每次 `openWith`）。
- [ ] 是否保留“回到源码编辑”的明确入口（命令、标题栏按钮、快捷键）。
- [ ] 在以下入口中是否都能直接落到 viewer：Explorer、Quick Open、最近文件、命令面板打开文件。
- [ ] 分屏/多窗口/拖拽标签时，是否不会退回普通文本编辑器。

### B. 兼容性与回归（必须）

- [ ] 与现有命令条件（`resourceLangId`/`activeCustomEditorId`）是否兼容，菜单显隐不混乱。
- [ ] Mermaid 注入链路（`markdown.previewScripts` / `markdown.previewStyles`）是否保持可用。
- [ ] 与常见 Markdown 扩展共存是否可接受（至少抽样：目录、lint、增强预览类）。
- [ ] Remote/WSL/多根工作区场景下行为一致。

### C. 用户体验（必须）

- [ ] 切换多个 Markdown 文件时，体感闪烁是否显著下降（目标：无“源码先闪一下”）。
- [ ] 用户从“阅读”切回“编辑”的成本是否可控（目标：<= 1 次操作）。
- [ ] 首次打开 Markdown 的心智是否清晰（不会让用户误以为“文件不可编辑/丢失”）。

### D. 性能与稳定性（建议）

- [ ] 首次打开与连续切换延迟是否可接受（建议记录 p50/p95）。
- [ ] 长文档、含多 Mermaid 图文档是否无明显卡顿或内存异常增长。
- [ ] 关闭/重开工作区后，editor association 状态是否可预测。

### E. 发布与回滚（必须）

- [ ] 先做 feature flag（默认关闭），仅灰度给愿意试用的用户。
- [ ] 提供一键回退到当前方案的开关与文档说明。
- [ ] 明确 Go/No-Go 门槛：若出现高频兼容性问题或编辑路径阻塞，则不发布默认开启。

### 建议验证顺序

1. 用最小 Spike 验证 A 组与 C 组（先看“真能消闪”）。
2. 通过后再做 B 组兼容性矩阵抽测。
3. 最后评估 D/E，给出 Go/No-Go 决策记录。

## 1 周 Spike 执行计划（高成本路线）

> 周期：5 个工作日（D1-D5）  
> 范围：只做可行性验证，不做完整产品化。

### D1（机制打样）

- 目标：最小化实现 `.md -> custom readonly viewer` 打开链路（feature flag 默认关闭）。
- 产出：
  - 新增开关配置（实验态）。
  - 最小 custom editor/provider 打桩实现（可打开 Markdown）。
  - 保留“回到源码编辑”命令入口。
- 当日验收：
  - 开关关闭时，行为与当前版本一致。
  - 开关开启时，从 Explorer 打开 `.md` 可进入 viewer。

### D2（入口与交互闭环）

- 目标：覆盖主要打开入口并补齐用户可逆路径。
- 产出：
  - 验证 Explorer / Quick Open / 最近文件 / 命令面板打开文件的落点一致性。
  - viewer 标题栏增加“切回编辑”按钮（或等效命令可见入口）。
- 当日验收：
  - 4 类入口均可直接进入 viewer。
  - 从 viewer 回到源码编辑 <= 1 次操作。

### D3（闪烁与性能采样）

- 目标：验证“最难路线”在核心体验上的收益。
- 产出：
  - 设计固定操作脚本（例如 10 个 md 文件连续切换 3 轮）。
  - 记录体感与时间指标（首次打开、连续切换，p50/p95）。
  - 对比基线（当前 `openWith` 切换方案）。
- 当日验收：
  - 主观上不再出现“源码先闪一下”或显著减弱。
  - 连续切换性能不明显劣于基线（建议阈值：p95 不劣化 >20%）。

### D4（兼容性抽测）

- 目标：快速识别高风险回归面。
- 产出：
  - 抽测维度：常见 Markdown 扩展、分屏/多窗口、Remote/WSL、多根工作区。
  - 记录问题清单（阻塞/高/中/低）。
- 当日验收：
  - 无“阻塞级”问题（无法编辑、无法回退、频繁崩溃）。
  - 高风险问题有可行绕过或回滚策略。

### D5（决策与收口）

- 目标：形成可执行的 Go/No-Go 结论。
- 产出：
  - 汇总证据：收益、成本、风险、剩余工作量。
  - 决策建议：继续推进 / 降级为中低成本路线 / 暂停。
  - 若继续：列出下一阶段里程碑与最小发布策略（灰度+回滚）。
- 当日验收：
  - 文档化结论完整，可被团队复核。
  - 明确默认发布策略（默认关闭，灰度验证后再议默认开启）。

## Spike 验收口径（统一）

- 体验收益：快速切换 Markdown 时，闪烁问题主观显著改善。
- 编辑可逆：任何时刻可快速返回源码编辑，不造成“被锁只读”感。
- 兼容底线：无阻塞级回归；高风险项有明确兜底（开关回退）。
- 发布约束：仅在 feature flag 下灰度，不直接替换默认打开行为。
